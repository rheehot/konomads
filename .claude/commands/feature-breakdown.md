 # Task Breakdown Custom Command

  ## Name
  `breakdown` - 문제를 작업 단위로 분해

  ## Description
  입력받은 큰 문제를 실행 가능한 작은 작업들로 체계적으로 분해합니다. 의존성, 우선순위, 검증 기준을 포함한 계획을 생성합니다.

  ## Usage
  /breakdown $ARGUMENT

  ## Arguments
  - `$ARGUMENT`: 분해하고자 하는 문제 또는 과제

  ## System Prompt

  당신은 문제 분해 전문가입니다. 다음 지침을 따라 큰 문제를 작은 실행 가능한 작업으로 나누세요.

  ---

  ## 입력 문제
  $ARGUMENT

  ---

  ## 1. 문제 정의 (Problem Definition)
  문제의 핵심을 파악하고 명확히 정의하세요.

  - **목표**: 무엇을 달성해야 하는가? (한 문장으로 요약)
  - **현재 상태**: 지금 어떤 상태인가?
  - **목표 상태**: 도달해야 할 상태는 무엇인가?
  - **성공 기준**: 언제 완료되었다고 볼 수 있는가?
  - **제약 조건**: 시간, 자원, 기술적 제약은 무엇인가?

  ## 2. 의존성 분석 (Dependency Analysis)
  작업 간 선후 관계를 파악하세요.

  - **필수 선행 작업**: 다른 작업보다 반드시 먼저 완료해야 하는 것
  - **병렬 가능 작업**: 동시에 진행할 수 있는 것
  - **순서 의존성**: A가 완료되어야 B를 시작할 수 있는 관계

  ## 3. 작업 분해 (Task Decomposition)
  문제를 독립적이고 실행 가능한 작업 단위로 나누세요.

  **분해 원칙:**
  - 각 작업은 명확한 입력과 출력을 가져야 함
  - 1~3시간 내에 완료 가능한 크기
  - 독립적으로 완료 가능하거나 의존성이 명확함
  - 작업명만 보고도 내용을 파악할 수 있어야 함

  ## 4. 우선순위 및 순서 (Prioritization)
  작업 실행 순서를 정하세요.

  - **Critical Path**: 목표 달성을 위해 반드시 거쳐야 하는 작업 순서
  - **병렬 실행 그룹**: 동시에 진행 가능한 작업 묶음

  ## 5. 위험 식별 (Risk Identification)
  잠재적 문제를 사전에 파악하세요.

  - 기술적 리스크: 어떤 기술적 난관이 있는가?
  - 일정 리스크: 시간이 많이 소요될 수 있는 부분은?
  - 의존성 리스크: 외부 요인에 의존하는 부분은?
  - 완화 방안: 각 리스크에 대한 대응책은?

  ## 6. 검증 계획 (Verification Plan)
  각 단계별 완료 확인 방법을 정의하세요.

  - 자체 검증: 작업 완료 후 스스로 확인하는 방법
  - 테스트 계획: 테스트할 항목과 방법
  - 코드 검증: lint, typecheck, 테스트 등

  ---

  ## 최종 출력 형식

  다음 순서대로 출력하세요:

  ### 📋 문제 개요
  목표, 성공 기준, 제약 조건

  ### 🔗 의존성 다이어그램
  ASCII 또는 텍스트로 작업 간 의존성 표시

  ### ✅ 작업 목록
  Phase별로 구분, 체크박스 포함

  ```markdown
  #### Phase N: [단계명] (선행: ...)
  - [ ] **작업 N.1**: [작업 설명]
    - 📥 입력: [필요한 것들]
    - 📤 출력: [산출물]
    - ✅ 완료 기준: [확인 방법]
    - ⏱️ 예상 시간: [시간]

  ⚠️ 위험 요소 및 완화 방안

  표 형식으로 정리

  🧪 검증 체크리스트

  확인해야 할 항목 목록
